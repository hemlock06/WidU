# `benchmark_transmission_rate.py`

> 전송 레이트 강건성 — "제안된 전송 스펙(50Hz 배치)이면 지금 성능이 유지되나?"

## 무엇을 / 왜

WIDYU 앱 전송 구상(워치 IMU 1초/회·폰 IMU 3초/회 배치)에서 **관건은 전송 간격이 아니라
'배치 안 샘플레이트'**다. WidU 낙상 모델은 50Hz(`L2.FS`)에서 학습·평가됨. 디바이스가
진짜 50Hz를 주면 성능이 유지되는지, 미만이면 얼마나 깨지는지 측정.

## 어떻게

- 배포와 동일: 모델은 **깨끗한 50Hz로 학습**(GroupKFold 피험자 분리).
- 테스트 입력만 **레이트 R로 열화**: 50→R 안티앨리어스 다운샘플 → R→50 복원(= R/2Hz 위
  정보 손실). 디바이스가 R Hz로 줄 때 '배포 모델이 받는 입력'을 모사.
- UMAFall 동기 손목·허리에서 워치/폰/둘다(OR) recall·오탐·F1을 R∈{50,25,12.5}Hz로 비교.

## 결과 (UMAFall 617 trial, 피험자 분리)

| 레이트 | 워치만 recall/F1 | 폰만 recall/F1 | 둘다 recall/F1 |
|---|---|---|---|
| **50Hz (스펙)** | **0.984 / 0.899** | **0.984 / 0.987** | **0.995 / 0.900** |
| 25Hz | 0.910 / 0.873 | 0.984 / 0.989 | 0.995 / 0.915 |
| 12.5Hz | 0.720 / 0.775 | 0.974 / 0.987 | 0.989 / 0.930 |

- **50Hz 행 = 기존 device 벤치마크와 완전 동일 → 스펙 충족 시 성능 손실 0**(모델이 학습한 바로 그 레이트).
- **워치(손목)가 레이트에 가장 민감**: recall 0.984→0.910→0.720. 손목 낙상은 빠르고 미묘 → **50Hz 바닥선이 워치에서 가장 빡빡**.
- **폰(허리)은 강건**(12.5Hz도 0.974). **둘다는 폰이 받쳐 12.5Hz도 0.989** = 융합 안전 마진.
- **디바이스 함의**: 워치 **반드시 ≥50Hz**(미달 시 손목 recall 붕괴), 폰도 안전하게 50Hz.
- ⚠️ 한계: 샘플레이트 1축만. 충격창 배치 보존(버스트)·타임스탬프 연속성은 별개 필수(스펙 2·3).
  열화 시뮬은 실센서 안티앨리어싱 근사. UMAFall 내부(낙관·시뮬 낙상).
