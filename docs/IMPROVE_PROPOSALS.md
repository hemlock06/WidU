# WidU improve — 제안 (구현하지 않음 · 옵션·트레이드오프)

> 야간 무인 작업에서 **설계결정이 필요하거나 독립검증이 곤란해** 코드로 구현하지 않은 항목.
> 미검증 코드 잔류 금지 원칙에 따라 여기 옵션·트레이드오프만 정리한다.
> 짝 문서: 구현·검증 결과는 `docs/IMPROVE_SUMMARY.md`, 근거는 `docs/HANDOFF_ISSUES.md`.

## P0-1. 상태 영속성 + 멀티워커 (HANDOFF P0-1) — 설계결정

현재 전 사용자 상태(L1 baseline·L4 추세·L5 알림예산·self-check 대기)가 프로세스 메모리
dict 에만 존재 → 재시작 시 리셋, 멀티워커 시 판정 비결정.

| 옵션 | 장점 | 단점/트레이드오프 |
|---|---|---|
| A. 외부 저장소(Redis/DB)로 사용자 상태 이전 | 멀티워커·재시작 안전, 수평확장 | 직렬화 스키마 설계·마이그레이션·지연·인프라 비용. 상태 객체(EWMA/추세) 직렬화 계약 필요 |
| B. 사용자→단일워커 스티키 라우팅 | 코드 변경 최소 | 워커 장애 시 상태 유실·재균형 어려움. 진정한 영속성 아님 |
| C. 주기적 스냅샷(파일/DB) + 기동 시 복원 | 콜드스타트 완화, 단순 | 스냅샷 간격만큼 유실 창. 동시성 락 필요 |

권고: 우선 **C(스냅샷 복원)로 콜드스타트 리셋 완화** → 확장 단계에서 **A**. 어느 쪽이든
공유 dict 동시성 락 검토 동반. WSGI(gunicorn/uwsgi)는 워커당 상태 전략 확정 후 도입.

## P0-2. 인증·인가 (HANDOFF P0-2) — 보안 설계결정

모든 엔드포인트 무인증 → 임의 `uid` 의 PII POST/조회 가능.

| 옵션 | 비고 |
|---|---|
| 디바이스 키(단말당 비밀) + `uid` 소유권 검증 | 단말-서버 모델에 단순·견고. 키 발급/회수 체계 필요 |
| JWT(단기 토큰 + 갱신) | 표준·확장 용이. 발급 서버/키 회전 운영 부담 |
| 공통: TLS 강제 + 레이트리밋 + 감사로그 | 전송보안·남용 억제 |

권고: 디바이스 키 + uid 소유권 검증부터, TLS·레이트리밋 동반. **본인(hemlo) 승인 후** 구현.

## P1-6. 서버→클라이언트 알림 전달 (HANDOFF P1-6) — 전달 설계결정

`escalation`(guardian_push/auto_call)이 문자열로만 반환, 실제 전송 없음.

| 옵션 | 트레이드오프 |
|---|---|
| FCM/APNs 푸시 + 보호자 등록 모델 | 제품 목적 충족. 토큰 관리·플랫폼별 연동·실패 재시도 필요 |
| 프론트 폴링(`GET /status`) 임시 | 최소 구현·즉시 가능. 지연·배터리·실시간성 한계 |
| 웹소켓/SSE | 실시간. 연결 상태 관리·서버 자원 |

권고: 임시로 폴링 계약 합의 → 정식은 FCM/APNs + 보호자 등록(영속성 P0-1 선행).

## P2-5. `RecordKind.SLEEP`/`STEPS` 무반응 (HANDOFF P2-5) — 의도 확인 필요

두 종류는 입력 수용되나 L4 탐지 로직 없음(`{"status":"ok"}`만).

- **의도된 미구현이면**: `types.py`/`l4_trend.py` 에 "수집만, 탐지 미정의" 주석·문서 명시.
- **탐지 의도면**: 수면시간 급감·걸음 급감(쇠약/낙상 후 활동저하 신호) 추세 규칙 추가 — 단,
  임상 임계·개인화 설계가 필요(추측 금지) → 별도 스펙.

권고: 우선 **문서로 "수집만" 명시**(소비측 오해 방지), 탐지는 임상근거 확보 후.

## 추가 제안 (이번 검증 과정에서 도출)

### A1. P1-2 입력검증의 더 외과적인 대안 (관측성 보존)

구현본은 전역 `errorhandler(KeyError/ValueError/TypeError)` → 400. 정상 입력엔 무영향이나,
**내부 로직의 진짜 500급 오류도 400 으로 가려질 수 있음**(모니터링/디버깅 저하). 대안:

- 각 핸들러 진입부에서 **명시적 경계 검증 헬퍼**(`_require(d, "bpm")`, enum 파싱 try)로
  잘못된 *요청*만 `werkzeug.exceptions.BadRequest`(400)로 올리고,
- 그 외 예외는 그대로 전파 → 별도 `errorhandler(Exception)` 가 **500 + JSON** 으로 처리.

트레이드오프: 핸들러별 코드 증가(검증 표면 확대) ↔ 400/500 의미 정확. 스키마 검증
라이브러리(pydantic/marshmallow) 도입 시 자연스럽게 해소.

### A2. 의존성 핀의 Python 3.9 호환 전략 (P1-5 보완)

`requirements.lock` 은 3.11 검증 핀(numpy 2.4 등 3.9 미지원). 현 CI 매트릭스(3.9·3.11)에서
3.9 잡은 범위 `requirements.txt` 로 설치 → 비결정 잔존.

| 옵션 | 트레이드오프 |
|---|---|
| 3.9 지원 중단(`requires-python>=3.11`) + 단일 lock | 가장 단순·완전 결정. 3.9 환경 포기(배포 타깃 확인 필요) |
| 3.9용 별도 lock(`requirements-py39.lock`, 구버전 핀) | 두 환경 모두 결정. 두 핀 유지보수·검증 필요 |
| `==` 대신 호환 범위(`>=,<상한`) + Dependabot | 설치 유연·메이저 폭주 차단. 완전 결정은 아님 |

권고: **배포 타깃 Python 확정**이 선행. 3.11+ 단일화 가능하면 옵션1(가장 단순·결정적).

### A3. 버전 단일화 마무리 (P2-1 보완)

`__init__.__version__` 은 메타데이터 우선·폴백 0.20.0 으로 일치시킴. 폴백 문자열이 여전히
별도 출처이므로, 장기적으로 **빌드시 주입**(setuptools-scm 등) 또는 폴백 제거(설치 강제)로
완전 단일화 가능. 트레이드오프: 소스 직접 실행(미설치) 편의 ↔ 단일출처 엄밀성.
